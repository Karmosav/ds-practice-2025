# System Model

The online bookstore is implemented as a client-server microservice system. The client (browser frontend) communicates synchronously with the orchestrator over REST/HTTP for checkout, and the orchestrator communicates with backend services over gRPC. For each order, the orchestrator generates a unique order ID, initializes transaction verification, fraud detection, and suggestions, and starts the checkout flow. Event ordering is handled with vector clocks across the three core services, with a partial order that includes concurrent validation events and dependent fraud/suggestion events (a, b, c, d, e, f). If any validation or fraud step fails, the flow is finalized as rejected; if all checks pass, suggestions finalizes the flow as approved and returns recommendations.

After approval, the orchestrator enqueues the order into a dedicated order queue, and executor replicas process orders under mutual exclusion using lease-based bully-style leader election (highest alive executor ID based on heartbeats). Only the current leader can dequeue orders. The system uses synchronous request/response on the checkout path for immediate user feedback and asynchronous behavior for internal worker threads and background order execution. Failure assumptions are mainly crash-stop and omission failures: services recover via container restarts, executors use retry/backoff when queue RPC calls fail, and leader leases expire automatically to enable re-election. Service state is in-memory, and cleanup is explicitly coordinated by the orchestrator via final vector-clock validation (each service clears local order state only when local_vc <= final_vc).
